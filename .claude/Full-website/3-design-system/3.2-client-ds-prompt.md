### Section 1: Project overview
One paragraph: what this site is, who it's for, what it must communicate in the first 5 seconds. This is the north star Claude Code reads before touching any component.

### Section 2: Color tokens
Every color as a CSS custom property.
**Variable Naming Standard**: All CSS custom properties must follow a strict layered prefix structure matching the design tokens database hierarchy:
- Core primitives: `--core-color-base-[name]` (e.g., `--core-color-base-white`)
- Palette colors: `--palette-color-[hue]-[shade]` (e.g., `--palette-color-stone-100`, `--palette-color-lime-500`)
- Brand specifics: `--brand-color-[concept]-[shade]` (e.g., `--brand-color-primary-500`, `--brand-color-accent-500`)
- Semantic mappings: `--semantic-color-[category]-[element]-[state]` (e.g., `--semantic-color-background-default`, `--semantic-color-foreground-neutral-default`, `--semantic-color-border-divider-subtle`)

Include:
- All background layers (minimum 3: base, raised, surface)
- All text colors (primary, secondary, muted)
- Primary accent with all variants (base, dim, border, hover)
- Secondary accent if applicable
- All border colors
- Status colors: success, warning, error
- Semantic assignments: what each color is used for
- **Contrast ratio for every text/background and UI-element/background pairing, computed and stated next to the pairing** (e.g. `--semantic-color-foreground-default on --semantic-color-background-default: 5.2:1 — pass`). Hard floors, non-negotiable: **4.5:1** for body text and any text under 24px regular / 19px bold, **3:1** for large text (24px+ regular or 19px+ bold), **3:1** for non-text UI elements (icon strokes, input borders, focus rings) against their background — WCAG 2.1 AA, SC 1.4.3 and SC 1.4.11. If the approved visual direction produces a pairing below these ratios, darken or lighten the token by the minimum amount needed to pass and flag the change ("Adjusted from approved direction to meet contrast floor: ...") instead of shipping a failing pair.

Format as a CSS `:root {}` block that can be copied directly into a stylesheet.

### Section 3: Typography
- Google Fonts import link (exact URL with only the weights needed)
- CSS font-family variables
- Complete type scale as CSS custom properties: display sizes with clamp(), heading sizes, body sizes, label sizes, mono sizes
- Line-height values per context
- Letter-spacing values per context
- Rules: when to use each font, which sizes are for which elements, forbidden combinations
- **Minimum font-size floor (hard floor, do not go below regardless of visual direction):** body copy and any paragraph/label read continuously never below **16px**; secondary/caption text never below **12px**, and only between 12–15px when paired with a contrast ratio of 4.5:1 or higher. This is an industry best-practice convention (Google web.dev, Material Design, and Apple HIG all converge near 16px for body text) — WCAG itself does not set a literal pixel minimum, it governs contrast (SC 1.4.3) and 200% resize support (SC 1.4.4) instead. State this floor explicitly in the type scale so Claude Code cannot default to 8–13px "secondary" text, which is the single most common accessibility failure in AI-generated sites.

### Section 8: Layout system
- Container class rules
- Standard grid configurations with CSS (2-col, 3-col, feature alternating)
- Breakpoints and what changes at each one
- **Mobile-first is mandatory, not a declaration to choose.** Write base CSS rules for the mobile viewport with no media query, then add complexity for larger screens using `min-width` media queries only. Never write desktop as the base styles with `max-width` media queries stripping things down for mobile — that pattern is exactly how conversion-critical elements (a hero CTA, in one production case) end up needing a late `!important` override to reappear on mobile instead of being correct there by default. State this rule explicitly in the DS so Claude Code cannot default to desktop-first structure. Two real, current reasons this is non-negotiable, not stylistic preference: Google's mobile-first indexing is 100% complete since July 2024 (confirmed on Google's own Search Central blog) — the mobile render of the page is the canonical version used for ranking, full stop, regardless of desktop traffic share. And mobile accounts for just over half of global web traffic — StatCounter and DataReportal's Digital 2026 report both independently put it around 51-52% as of 2026 — the base case is mobile, desktop is the enhancement, not the other way around.

- **Accessibility floors — these override every other rule on this list, including the visual direction, if they ever conflict:** no body text below 16px; no text of any kind below 12px; no text/background contrast below 4.5:1 (3:1 for large text 24px+/19px+ bold); no icon, border, or focus-ring contrast below 3:1 against its background; no clickable target smaller than 24×24px (44×44px for primary CTAs), with at least 24px of center-to-center spacing from the next target; no `outline: none` (or equivalent) without a visible replacement focus state. Source: WCAG 2.1 AA SC 1.4.3 / 1.4.11, WCAG 2.2 AA SC 2.5.8, WCAG 2.2 AAA SC 2.5.5 (target size, recommended for primary CTAs), WCAG 2.1 AA SC 2.4.7 (focus visible). ## CRITICAL AI GUARDRAILS (HALLUCINATION-PROOFING)

To prevent AI hallucinations and generic "slop" output, you must strictly follow these rules:
1. **No Code Stubs**: You must provide complete CSS variables, styles, and Javascript scroll observers. Do not write ellipses (`...`) or placeholder comments (e.g. `/* other rules here */`) in the code blocks. They must be drop-in ready.
2. **Grounding in Approvals**: Do not invent new colors, typography, or styling guidelines that deviate from the approved visual direction context provided.
3. **No AI Copy Patterns**:
   - Do not use em dashes (`—`) in copy.
   - Do not use emojis in written body text or headlines.
   - Do not use SaaS clichés ("Not only X, but Y", "Transform your business").
4. **Accessibility Minimums (Non-Negotiable)**: Every color, typography, and layout token defined in Sections 2, 3, and 17 must satisfy the accessibility floors stated there: contrast ≥4.5:1 for body text and ≥3:1 for large text (24px+/19px+ bold) and non-text UI elements (WCAG 2.1 AA, SC 1.4.3 / SC 1.4.11); body text ≥16px, no text anywhere below 12px (industry best practice, not a literal WCAG pixel rule); clickable targets ≥24×24px, ≥44×44px for primary CTAs (WCAG 2.2 AA SC 2.5.8, AAA SC 2.5.5); visible focus state on every interactive element (WCAG 2.1 AA SC 2.4.7). **These floors override Rule 2** if the approved visual direction would otherwise violate them — adjust the token and state the deviation explicitly ("Adjusted from approved direction to meet accessibility floor: ...") rather than shipping an inaccessible value. Do not invent numbers beyond what is cited here; if a contrast ratio cannot be computed exactly, state the formula used.
5. **Mobile-First CSS (Non-Negotiable)**: Section 8's layout system and every component section (9-16) must be written with mobile as the base, unmodified state and larger breakpoints added via `min-width` media queries. Never generate desktop-first CSS (`max-width` overrides stripping a desktop layout down for mobile) — this exact pattern was confirmed on a real client site, where a hero CTA positioned `absolute` by default needed a late `!important` override at a `max-width` breakpoint just to work correctly on mobile, instead of being correct there by default. The override happened to work in that case; relying on that is still the fragile, wrong-order way to build it. If the visual direction input describes a desktop-oriented layout, translate it into a mobile base + progressive enhancement structure rather than copying it as the default state.
