# Step 7.1 — Inventário de Componente

Você está analisando um componente de UI de um site entregue para mapear seus valores específicos do cliente aos tokens semânticos do design-system VellumWire.

## O QUE ENTREGAR

**1. Tabela de mapeamento cliente → token semântico**

| Propriedade CSS | Valor do cliente | Papel semântico | Token `--semantic-*` candidato |
|---|---|---|---|
| (cada valor específico do cliente) | (hex, px, etc.) | (cor primária, espaçamento base, etc.) | (a preencher com o grep do próximo passo) |

*Valores já em `var()` do projeto do cliente: documentar como `var(--client-X)` na coluna "Valor do cliente" e identificar o papel semântico equivalente no DS.*

**2. Comandos grep para buscar os tokens semânticos relevantes**

Com base nos papéis identificados na tabela acima, gere os comandos grep exatos para rodar em `dist/css/tokens/brand-vellumwire.css`. Inclua só as categorias que o componente realmente usa:

```bash
# Cores (só se o componente usa cor)
grep "semantic-color" dist/css/tokens/brand-vellumwire.css

# Espaçamentos e tamanhos
grep "semantic-size" dist/css/tokens/brand-vellumwire.css

# Tipografia
grep "semantic-type" dist/css/tokens/brand-vellumwire.css

# Motion
grep "semantic-motion" dist/css/tokens/brand-vellumwire.css
```

**3. Slots e variantes identificados**

Liste os `data-*` atributos e slots opcionais que já existem ou que você sugere estruturar no componente migrado. Será usado no `spec.md` do próximo passo.

**4. Gaps** (se houver)

Se um papel semântico claramente não tem correspondência nos tokens existentes (ex: uma cor de destaque fora do padrão do DS), sinalize aqui. Esses gaps precisam ser resolvidos em `docs/tokens-architecture.md` antes do Passo 2.
