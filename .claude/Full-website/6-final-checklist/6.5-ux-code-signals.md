# Step 6.5 — UX, Code & Human Signal Reference

Manual reference for thorough mode. Maps to U1–U3, K1–K2, H1.

---

## UX Logic Signals (U1–U3)

### U1 — Navigation shells
Menus and dropdowns built as visual structures with no real destinations behind them.
- Symptoms: dropdown items linking to `#` or empty, mobile hamburger opening with no real navigation items, footer columns with placeholder links
- Fix: every visible navigation item must resolve to a real destination before the page ships. `href="#"` is a Blocker.

### U2 — Missing form states
Forms that exist visually but have no defined error, empty, loading, or disabled behavior.
- Symptoms: form with no error messages or validation feedback, button with no hover/active/focus/disabled state in CSS, modal with no close button or Escape key behavior, empty data view with no empty state
- Fix: every interactive component needs minimum four states: default, hover, active/pressed, disabled. Forms need error and success. Data views need empty and loading.

### U3 — SaaS hero formula
(Same as D7 checklist item — flagged here as a UX pattern, not just visual.) If all four are present simultaneously: centered badge with emoji, large centered h1, two symmetrical CTA buttons, product screenshot below. Flag only when all four coexist.

### Flow inconsistency across pages
The AI designs individual screens well but fails at IA across the site. Each page polished in isolation, connections broken.
- Symptoms: no clear path back to the homepage, breadcrumbs on some pages and not others with no logic, footer nav doesn't match header nav
- Fix: map every page-to-page connection before building. Every page needs one path back to where it came from and one path forward.

## Code Signals (K1–K2)

### K1 — Console.log in production ⚠️ Blocker
`console.log()` statements from the generation process left in production. Logs with emojis are a hard launch blocker.
- Symptoms: `console.log('🚀 Component mounted!')` / `console.log('✅ Form submitted successfully')` / `console.log('data:', data)`
- Fix: grep for `console.log` before every deploy. In production, logs should not exist or route through a logging service.

### K2 — Explanatory comments
Comments in HTML, CSS, or JS describing what the code does instead of why a decision was made.
```html
<!-- Contact section of the website -->
<!-- Main navigation bar -->
```
```js
// Function to open the mobile menu
// Loop through all items
```
A human developer comments when the *why* is non-obvious: a workaround, a browser bug, a constraint. The AI comments to explain what any reader can already see.
- Fix: delete every comment that describes what. Keep only comments that explain why.

### Redundant JS logic
Simple UI tasks (toggling a class, showing/hiding) implemented through unnecessarily complex conditionals.
```js
// AI version — 8 lines
function openModal() {
  const modal = document.getElementById('modal');
  if (modal !== null && modal !== undefined) {
    if (modal.style.display === 'none' || modal.style.display === '') {
      modal.style.display = 'flex';
    }
  }
}
// Should be: modal.classList.toggle('is-open') + CSS
```
Fix: prefer CSS state management (`.is-open`, `.is-active`) over inline style manipulation. If basic UI interaction needs more than 3–4 lines, question whether CSS can own it.

## Missing Human Signal (H1)

These are conditions to check, not patterns to remove. If none are true of the page, it reads as AI regardless of everything else.

### H1a — Something only the client's team would know
A detail, number, or decision not in the brief, the competitor analysis, or any generic industry description. A constraint they worked around. A product decision that surprised them. Something requiring having actually spoken to the people who built it.

### H1b — An acknowledgment of what the product doesn't do
"We don't solve X. That's a different problem." Restraint about scope is more convincing than a claim of completeness. The AI never admits a gap. Humans who know their product do.

### H1c — A specific decision moment
Not "we considered several approaches" but "we tried X. It failed because Y." Failure requires specificity. The AI has no failed attempts to report.

### H1d — One moment of genuine uncertainty
"We're not sure this works for teams above 50 people." Hedging is AI. Specific uncertainty — naming the condition under which a claim might not hold — is human.

---

If none of H1a–H1d exist anywhere on the page: the page reads as AI-generated regardless of how clean everything else is.
