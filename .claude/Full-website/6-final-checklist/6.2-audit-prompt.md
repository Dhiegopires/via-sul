# Step 6.2 — AI Self-Audit Prompt

Paste into a **new Claude chat** — not the Claude Code session that built the page. Fresh context is the point. Paste the full HTML of one page into the `[paste here]` field before sending.

Run one page per chat. Do not batch multiple pages.

---

```
You are auditing a web page for AI-generated patterns in copy, design, and code. Find every instance of the patterns below. Report each with page location, exact quote or code snippet, severity level, and fix.

Be specific. "The hero subtitle sounds generic" is not a finding. "The hero subtitle reads 'A complete platform for collaboration': this phrase applies to any SaaS product and contains no information specific to this client" is a finding.

Do not flag things outside this list. Do not compliment the page. Only report violations.

PAGE HTML:
[paste the complete HTML here]

---

COPY PATTERNS:

C1: "Not X, but Y" contrast trap — flag every negative/positive contrast pair. Max 1 per page.
C2: Short sentence punch overuse — flag standalone short declarative sentences used for emphasis. Max 1 per section.
C3: Repeated statistics — flag any number or percentage appearing more than once.
C4: Repeated key phrases — flag any phrase used more than once across different sections.
C5: "That's by design" — flag: "That's by design", "And that's intentional", "This is a deliberate choice", "Not by accident", "By design, not default."
C6: Meta-commentary — flag sentences that describe what a section is doing instead of doing it. "Much of the work lives in...", "The goal of this section is to..."
C7: Generic filler — flag: "close the loop", "without overwhelming the user", "straightforward path forward", "fits into existing workflows", "make informed decisions", "actionable insights", "seamless experience", "at every breakpoint", "stay coherent", "end to end from first sketch to final pixel", "full design surface", "solo or embedded in product teams", "details that make products feel alive", "built to scale without breaking", "work as hard as they look."
C8: Three-item adjective lists — flag any list of exactly three generic adjectives describing the same thing.
C9: Defensive framing — flag: "These patterns aren't invented here", "This isn't a novel concept", "We're not claiming to have solved X", "Obviously this won't work for every team."
C10: Generic CTAs — flag: "Discover More", "Start Your Journey", "Optimize Now", "Transform Your Business", "Explore the Possibilities."
C11: Soulless institutional copy — flag any company/mission description where swapping the name for a competitor's leaves the paragraph intact.
C12: Headline/subheadline disconnect — flag pairs where the subheadline does not answer what the headline set up.
C13: Fragmented headline poetry — flag headlines split across 3+ lines where reading as a continuous sentence reveals a generic claim.
C14: Content duplication — flag any topic appearing twice in different formats with no new information in the second pass.
C15: Generic testimonials — flag quotes with no specific result, number, or feature name requiring firsthand experience.
C16: Em dashes — flag every em dash (—) in copy. All of them. No exceptions.
C17: Emojis in copy — flag every emoji in headlines, body text, or CTAs. UI status indicators exempt.
C18: Placeholder text — flag: "lorem ipsum", "[CLIENT NAME]", "[company]", "insert your text", "placeholder", "coming soon" where real content was expected.
C19: Link text/destination mismatch — flag every link or button whose visible text promises a specific destination or action while its `href` points somewhere else (most commonly a "lower commitment" alternative link secretly pointing to the same destination as the primary CTA). Compare every link's text against its actual `href`, don't assume they match.
C20: Proof that doesn't answer "will this make me money" — flag any results/stats section where a technical or vanity metric (Lighthouse score, page-load time, time-on-page, clicks-per-step, "X CTA on the page") is given equal or greater prominence than an available lead/conversion number (form submissions, qualified consultations, clients closed). A real, accurate number can still fail this check if it doesn't tell a non-technical reader whether hiring this business will make them money.

---

DESIGN & CSS PATTERNS:

D1: Predictable gradients — flag CSS gradients with purple/indigo/violet/cyan as primary: #7c3aed, #6366f1, #2563eb or similar with no design system justification.
D2: Excessive border-radius — flag any border-radius above 8px on cards/buttons/inputs unless it's a pill (9999px) with clear semantic purpose. Do not flag values from CSS variables.
D3: Gratuitous shadows — flag box-shadow spread above 20px, or shadow-xl/shadow-2xl on cards where no elevation hierarchy exists.
D4: Colored left border — flag border-left applied uniformly to every card in a grid with a hardcoded hex color. Do NOT flag border-left referencing a CSS variable.
D5: Hover on non-interactive elements — flag transform/color transitions on elements with no href, button role, or onClick.
D6: Service list duplication — flag compact summary (list/marquee) + full expanded sections below with no new information in the expanded version.
D7: Sub-16px text and low contrast (Blocker) — flag font-size below 16px on body copy, below 12px anywhere, text/background pairs visibly under 4.5:1 (normal text) or 3:1 (large text/non-text UI), and outline:none/outline:0 with no visible replacement focus state. Also flag disabled buttons or form fields rendered via opacity alone with no HTML `disabled` attribute — opacity reduces contrast below threshold.
D8: Color-only state indicators (Blocker) — flag error, success, warning, or info states that communicate exclusively through color with no secondary signal (icon, text label, or shape). Example: red border on an invalid input with no error message text; green icon with no "Success" label. Each state must have at least one non-color differentiator.

---

UX PATTERNS:

U1: Navigation shells — flag href="#" or href="" on visible navigation links, dropdowns, or footer links.
U2: Missing form states — flag forms with no error state styling (.error class, :invalid styling, or error message element) in CSS.
U3: SaaS hero formula — flag if the hero has ALL FOUR simultaneously: (a) centered badge with emoji, (b) large centered h1, (c) two symmetrical CTA buttons, (d) product screenshot immediately below.

---

CODE PATTERNS:

K1: Console.log — flag every console.log. Highest severity if it contains an emoji.
K2: Explanatory comments — flag HTML/JS comments describing what the code does: "<!-- Contact section -->", "// Function to open the menu", "// Loop through items."

---

MISSING HUMAN SIGNAL:

H1: Check whether any of these exist anywhere on the page:
- A specific detail or decision only someone who did the work would know (not from a brief or competitor analysis)
- An acknowledgment of what was NOT solved or left out
- A moment of specific uncertainty ("we don't know if X works for Y use case")
If none exist: flag as MISSING HUMAN SIGNAL — page reads as AI-generated regardless of other results.

---

ACCESSIBILITY — KEYBOARD NAVIGATION:

A1: Skip link — flag if the first Tab stop is NOT a "skip to content" link (or equivalent) that jumps to the main content area. No skip link = Blocker.
A2: Tab order — flag if focus order is visually illogical: jumps to footer before body content, skips sections, or follows DOM order that doesn't match visual order.
A3: Keyboard trap — flag any component where pressing Tab, Shift+Tab, and Esc cannot exit. Modals that don't close with Esc are a Blocker.
A4: Icon buttons without ARIA label — flag `<button>` or `<a>` elements that contain only an icon (SVG, img, or icon font character) with no `aria-label`, `aria-labelledby`, or `<title>` inside the SVG. Blocker.
A5: Form fields without labels — flag any `<input>`, `<select>`, or `<textarea>` that lacks a `<label for>`, `aria-label`, or `aria-labelledby`. Placeholder alone is not a label. Blocker.
A6: Missing document language — flag if the `<html>` tag lacks a `lang` attribute (e.g. `lang="pt-BR"` or `lang="en"`). Missing `lang` is a WebAIM top-6 failure: screen readers default to the OS language and mispronounce every word. Blocker.

Note: full manual keyboard test (Tab through the entire page) is in `6.7-a11y-keyboard.md` — the above flags what static analysis can catch.

---

REPORT FORMAT — use exactly this structure for each finding:

ID: [C1 / D2 / U1 / K1 / H1 / etc.]
Severity: [Blocker / Cut / Rewrite / Limit / Add]
Location: [section name or line description]
Finding: [exact quote or code snippet]
Fix: [one specific action]

Severity:
- Blocker: broken functionality, placeholder text, console.logs, accessibility violations. Fix before any review.
- Cut: adds no information. Delete immediately.
- Rewrite: idea worth keeping, execution weak. Rewrite with specificity.
- Limit: acceptable once per page, currently over limit. Reduce to 1.
- Add: something missing that would make the page feel human.

After all findings: TOTAL: [n] findings ([n] Blockers, [n] Cuts, [n] Rewrites, [n] Limits, [n] Adds)
```
