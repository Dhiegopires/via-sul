# Step 6.4 — Design & Layout Signals Reference

Manual reference for thorough mode. Maps to D1–D8 (visual/CSS) and the layout/structure patterns.

---

## Visual & CSS Signals (D1–D8)

### D1 — Predictable gradients
Purple-to-blue or indigo-to-cyan gradients with no brand justification. The model defaults to this when no visual constraint is given.
- Symptoms: `background: linear-gradient(135deg, #7c3aed, #2563eb)`, gradient on hero + buttons + dividers simultaneously
- Fix: gradients need a reason. If the brand color isn't in it, it shouldn't be there. One gradient per page max.

### D2 — Excessive border-radius
Indiscriminate `border-radius: 16–24px` on every element — cards, inputs, containers — without understanding what roundness communicates.
- Symptoms: every element has the same radius, large containers with radius that clips content, pill buttons (`9999px`) on primary actions without brand rationale
- Fix: radius should be a design token with a defined system. Cards 8px, buttons 6px, pills reserved for tags/labels. Uniform 20px+ across everything is a generator default.
- **Exception:** values from a CSS variable are a design system decision, not this pattern.

### D3 — Gratuitous shadows and glows
Soft `box-shadow` (`shadow-xl`, `shadow-2xl`) and neon glows applied freely to simulate depth, resulting in everything feeling equally heavy.
- Symptoms: `box-shadow: 0 25px 50px rgba(0,0,0,0.25)` on every card, shadows stacked inside shadows, neon glow on CTAs with no state-feedback purpose
- Fix: shadows communicate elevation. Each elevation level gets exactly one shadow value. If every element has `shadow-xl`, nothing has elevation.

### D4 — Colored left border on cards
4px vertical `border-left` in purple or blue applied uniformly to every benefit/feature card. Identifiable on sight.
- Symptoms: `border-left: 4px solid #7c3aed` on every card in a grid, combined with icon above and two lines of text
- Fix: if an accent is needed, it should mean something. A border that's identical on every card communicates nothing.
- **Exception:** `border-left` referencing a CSS variable is a design system decision (e.g. status tokens with varying color per state). This pattern is decorative uniformity with a hardcoded hex.

### D5 — Hover on non-interactive elements
Microinteractions on static elements — cards without links, text blocks that scale on hover. The AI adds "interactive feel" without filtering for whether the element does anything.
- Symptoms: `transform: scale(1.02)` on a card with no `href`, glow on a decorative icon, color change on non-link text
- Fix: hover states indicate interactivity. If the element doesn't respond to a click, remove the hover effect.

### D6 — Service list duplication
A compact summary (list, marquee, grid) of services followed lower on the same page by full expanded sections with the same information and no new specifics.
- Fix: the expanded sections must contain information the compact list doesn't. Otherwise cut one.

### D7 — Sub-16px text, low contrast, and disabled-by-opacity ⚠️ Blocker
The single most common accessibility failure. Treat the same as a console.log or a broken link.
- Body text below 16px, anything below 12px anywhere
- `color: #999`, `#aaa`, `#bbb` on white/light backgrounds (typically below 3:1 contrast)
- Icon strokes, input borders, or focus rings barely visible against their background
- `outline: none` or `outline: 0` with no visible replacement focus state
- `opacity: 0.4` (or any reduced opacity) used as the sole signal for a disabled button — opacity alone drops contrast below 4.5:1 and is invisible to colorblind users

WCAG 2.1 AA minimums:
- 4.5:1 — normal text (SC 1.4.3)
- 3:1 — large text (24px+ / 19px+ bold) and non-text UI elements (SC 1.4.3 / 1.4.11)
- 24×24px minimum clickable target, 44×44px for primary CTAs (WCAG 2.2 AA SC 2.5.8)

Fix: verify every text/background pair with a contrast checker before shipping. `outline: none` must have a visible CSS replacement. Disabled buttons must use the HTML `disabled` attribute and include a visible tooltip or inline label explaining how to enable the action — not just lower opacity.

### D8 — Color-only state indicators ⚠️ Blocker
Error, success, warning, and info states that communicate exclusively through color fail users with color blindness (affects ~8% of males). WCAG 1.4.1 prohibits using color as the only visual means of conveying information.
- Symptoms: red border on an invalid input with no error message, green checkmark icon on success with no text, an "active" tab highlighted only by a blue underline
- Fix: every state that uses color must also use a second signal — an icon (✗ / ✓ / ⚠), text label, pattern, or shape change. The icon must have `aria-hidden="true"` and the state must be announced via `aria-describedby` or `role="alert"` so screen readers get it too.

## Layout & Structure Signals

### "Frankenstein" section rhythm
Sections that each feel complete but don't build on each other. The page reads like separate articles stitched together, not an argument.
- Detect: try to describe the narrative arc in one sentence. If you can't, the structure is AI-generated.
- Fix: each section either answers a question raised by the previous one or deepens a claim already made.

### Metric blocks as wallpaper
A visual stat/citation block repeated after every section regardless of whether it adds information.
- Fix: appears only when the link to the problem statement is non-obvious. If the section body already makes the connection, cut the block.

### Pullquotes that restate the paragraph
A pullquote that repeats the most quotable sentence from the paragraph above instead of adding a new angle.
- Fix: pullquotes earn their space by saying something the surrounding text doesn't.

### SaaS hero formula
All four elements simultaneously: (a) centered badge with emoji, (b) large centered h1, (c) two symmetrical CTA buttons, (d) product screenshot below. The statistical average of SaaS landing pages.
- Fix: break one element of the formula. Off-center alignment, a single CTA, a real screenshot with real data, or a non-centered headline all disrupt the pattern.

### Perfectly symmetrical grid sections
Benefit/feature sections with every card at identical height, text length, and icon size. Equal weight across everything means no judgment was applied.
- Fix: vary intentionally. One card gets more text because the idea needs more explanation. The grid reflects hierarchy, not uniformity.

### Decorative spacing
Excessive padding to fill space rather than create readable rhythm. Sections that feel "luxurious" but are actually thin content padded to avoid writing more.
- Symptoms: `padding: 120px 0` on minimal content, 200px+ hero padding pushing real content below the fold
- Fix: spacing driven by reading rhythm and visual grouping, not available space.

### Service list duplication (layout version)
A scrolling marquee or bullet list naming six services followed by six identical section blocks lower on the page with nearly the same descriptions reworded.
- Fix: keep only the version that carries more information. One pass per topic.
