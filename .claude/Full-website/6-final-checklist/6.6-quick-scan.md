# Step 6.6 — Quick Scan: Grep Targets + Severity

Run these before every final review. Each hit is a flag, not a guaranteed fix — check context before cutting.

---

## Severity Classification

| Level | When | Action |
|---|---|---|
| **Blocker** | Broken functionality, placeholder text, console.logs, accessibility violations (sub-16px body, sub-4.5:1 contrast, missing focus state, undersized touch targets) | Fix before any review |
| **Cut** | Pattern adds no information | Delete immediately |
| **Rewrite** | Idea worth keeping, execution weak | Rewrite with specificity |
| **Limit** | Acceptable once per page, currently over limit | Reduce to 1 |
| **Add** | Something missing that would make the page feel human | Write or design new content |

## Code quality grep targets ⚠️ Blockers

```bash
# CSS inline — deve retornar zero resultados
grep -rn 'style="' *.html **/*.html

# Vendor prefixes ausentes — se existir a versão padrão sem a prefixada antes, é Blocker
grep -n "backdrop-filter" assets/css/style.css | grep -v "\-webkit"
grep -n "mask-image" assets/css/style.css | grep -v "\-webkit"
grep -n "appearance:" assets/css/style.css | grep -v "\-webkit"

# min-height: 100vh sem fallback iOS
grep -n "min-height: 100vh" assets/css/style.css
# (verificar manualmente se a linha anterior é min-height: -webkit-fill-available)

# Tabindex positivo — deve retornar zero resultados
grep -rn 'tabindex="[1-9]' *.html **/*.html
```

```powershell
# CSS inline
Get-ChildItem -Recurse -Include *.html | Select-String -Pattern 'style="'

# Vendor prefixes ausentes
Get-Content assets/css/style.css | Select-String "backdrop-filter" | Where-Object { $_ -notmatch "-webkit" }
Get-Content assets/css/style.css | Select-String "mask-image" | Where-Object { $_ -notmatch "-webkit" }

# Tabindex positivo
Get-ChildItem -Recurse -Include *.html | Select-String -Pattern 'tabindex="[1-9]'
```

## Keyboard + document a11y grep targets ⚠️ Blockers (presença / ausência)

## Performance / production anti-pattern grep targets ⚠️ Blockers

```
cdn.tailwindcss.com
```

Any hit is a Blocker: this loads a client-side CSS JIT compiler in production, render-blocking, against the tool's own documented guidance. Compile to a static stylesheet instead. If a hit shares the page with a separate compiled `style.css`, that is a second, distinct Blocker — two competing styling systems loaded at once.

## Conversion tracking grep targets ⚠️ Blockers

```
generate_lead
```

For every hit: check what triggers it. If it's an `onclick` on a link to an external tool (Calendly, Typeform, WhatsApp) rather than a callback fired on that tool's actual completion event, it is a Blocker — the event name claims a confirmed lead but only proves a click. See `5.1-session-start-prompt.md` Conversion Tracking rules for the fix per tool.

## Structural checks (manual)

- [ ] Read only the first sentence of each body paragraph — do they all start the same way?
- [ ] Count "not X / but Y" contrasts — more than 1 per page?
- [ ] Find each statistic — used more than once on the page?
- [ ] Describe the page's narrative arc in one sentence — can you?
- [ ] Click every navigation link — all go somewhere real?
- [ ] Click every CTA and secondary link and compare its destination against its visible text — does it go where the words say it goes, not just "somewhere real"? Two differently-worded links pointing to the identical URL is the specific pattern to catch.
- [ ] If the hero has any JS-driven variant (carousel, segment picker, rotating cards), reload the page fresh and look at what actually renders after JS runs, before touching anything — is the approved CTA still there, or did the default state swap it for something else?
- [ ] Trigger every interactive element — error, empty, and loading states defined?
- [ ] Every form with name/email/phone fields links to a real, live privacy policy page (click it, confirm it's not a 404) — and if the site uses cookies/analytics beyond strictly necessary, a cookie consent banner appears before those scripts fire
- [ ] Is there anything on this page only someone who built it would know?
- [ ] Does any service or feature appear twice with no new information in the second pass?
- [ ] Read every section headline as one continuous sentence — specific claim or generic tagline?
- [ ] Run main text/background pairs through contrast checker — body ≥4.5:1, large text/icons ≥3:1 (WCAG 2.1 AA SC 1.4.3 / 1.4.11)
- [ ] Tab through every interactive element — focus state visible at every stop? (WCAG 2.1 AA SC 2.4.7)
- [ ] Check every clickable target at 375px — at least 24×24px? (WCAG 2.2 AA SC 2.5.8)

## Keyboard navigation checks (manual — ver 6.7 para detalhes)

- [ ] Pressionar Tab uma vez do topo da página — skip link aparece?
- [ ] Tab order percorre a página em ordem visual lógica?
- [ ] Todo modal/dropdown fecha com Esc e devolve foco ao elemento de origem?
- [ ] Consegue completar o percurso principal do site (home → CTA principal) só com teclado?

## Colorblind simulation (manual — ~2 min por página)

Abra a página no Chrome DevTools → Rendering → Emulate vision deficiencies. Testar:

- [ ] **Deuteranopia** (daltonismo vermelho/verde, tipo mais comum — ~6% dos homens): todos os estados de formulário (erro, sucesso, warning) legíveis sem a cor?
- [ ] **Protanopia** (vermelho cego): CTAs e links distinguíveis do texto base?
- [ ] **Achromatopsia** (escala de cinza): hierarquia visual e estados de formulário ainda comunicam o que precisam?

Se qualquer estado falhar: adicionar ícone + texto label além da cor (WCAG 1.4.1).
