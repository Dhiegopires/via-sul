# Step 6.8 — Conversion Architecture Regression Check

This is a drift check, not a design review. The decisions this file verifies — awareness level, persuasion formula, section sequence, CTA placement, attention ratio — were already made and signed off in Step 2.3 (conversion architecture) and Step 4.3 (layout preview, including the CTA fold check). If a page fails anything here, the fix is not "adjust it now" — it means the built page diverged from what was approved, and the question to ask is where in Step 5 that happened.

**Do not use this checklist as the first place conversion structure gets evaluated.** If a page reaches this step and nobody has looked at CTA placement or section order before now, that is itself the failure this file exists to catch — the process broke upstream, not here. This file cannot design a persuasion architecture retroactively; it can only confirm the one that was already approved actually shipped.

---

## What to check per page

**CV1 — Section order matches the approved layout preview**
Open the built page next to `layout-preview-[cliente].md`. Section by section, confirm the order is identical. Any section reordered, merged, split, or dropped during development without going back through Step 2.3 is a Blocker — not because the new order is necessarily wrong, but because it was never approved.

When a fix instruction says to relocate a section (e.g. "move the results to the top"), verify the entire named block actually moved — grep for it, confirm the old location is empty and the new one has the full content, don't just confirm *a* stat or headline resembling it appears near the top. A partial move (one representative number duplicated into the hero while the full section stays buried where it was) technically responds to the instruction's words without doing the thing asked — see C20 in `6.3-copy-signals.md` for the content-selection half of this same failure.

**CV2 — Primary CTA fold check, verified in the real browser, in its actual default state, mobile-first**
Open the live page fresh (hard refresh, don't interact with anything first). At 375px first — this is the check that matters most, run it before desktop — confirm the primary CTA (or the approved lower-commitment secondary CTA, if the page's awareness level required agitation first) is visible without scrolling. Then confirm the same at a common desktop width. This was already predicted in the Step 4.3 wireframe using approved copy — this check confirms the prediction held once real fonts, images, and browser rendering are involved.

**If the hero has any JS-driven variant state** (a rotating card carousel, an audience/segment picker, a tab switcher), check what renders by default the instant the page finishes loading — not a state reached only after the visitor interacts with something. A hero that hides the approved CTA behind a different action in its default state (e.g. an audience-segmentation choice replacing the demo CTA on load) fails this check even if the "real" CTA technically exists somewhere in the DOM — a visitor who never clicks the carousel never sees it. Confirmed production example: a hero carousel's first card set `segBtn: true`, which on load hid the primary CTA and secondary link entirely and showed only two audience-segment links instead, routing every default visitor to a different page before any CTA was reachable.

If any of this fails: Blocker. Real users will never reach a CTA that isn't actually there by default.

**CV3 — Attention ratio, counted on the shipped page**
Count every link, button, or CTA competing with the primary CTA within the mandatory section sequence (nav items in the hero viewport, secondary CTAs, unrelated outbound links). Compare against the budget approved in Step 2.3. Flag if development introduced any competing action that wasn't accounted for — a "share this page" button, an extra footer CTA, a chat widget that pops up over the hero.

**CV4 — CTA repetition cadence**
Scroll the full page. Confirm no more than 3 consecutive sections pass with zero visible next step, per the cadence rule set in Step 2.3.

**CV5 — Drop-off objection coverage**
Cross-reference Step 2.2's named drop-off risk and Step 2.3's assigned section for addressing it. Confirm that section, as built, actually names and defuses the objection — not a generic reassurance line that lost the specific objection somewhere in copywriting or editing.

**CV6 — Every CTA's destination matches its text (link honesty)**
Click every CTA and every secondary/alternative link on the page, including the ones offered as a lower-commitment escape from the primary ask. Compare where it actually goes against what its visible text promised. Pay specific attention to any link phrased as an alternative to the primary CTA ("prefer to do X instead?") — confirm it goes to a genuinely different destination that does X, not a copy-pasted primary CTA link with new wording on top. A visitor who clicks an "alternative" and lands on the identical destination as the button they were trying to avoid loses trust in the page immediately, and that loss is invisible in analytics — it just shows up as a session that leaves without converting.

**CV7 — Complete the conversion for real, end to end, before launch**
Every other check in this file verifies that the page *presents* correctly. None of them prove that completing the action actually produces a lead on the business side. Before launch, someone must actually do the thing a visitor would do — book a real slot on the live Calendly link, submit the real lean fallback form with test data, send the WhatsApp message — and confirm, concretely, that it arrives somewhere a human at the client will see it: the client's calendar shows the booking, the inbox that receives form submissions actually got the email (check it isn't in spam), the WhatsApp number actually receives the message. A page can pass CV1–CV7 perfectly and still generate zero real leads if the booking calendar has no available slots, the form's notification email silently fails, or the WhatsApp number is wrong or unmonitored. This is not a redundant step — it is the only check in this entire methodology that verifies the business, not the page, actually receives what the page promises to deliver. Cancel or delete the test booking/message afterward so it doesn't pollute the client's real data.

---

## Severity

All seven (CV1–CV7) are Blockers, same tier as a console.log or a missing skip link. A page with clean copy, clean code, and perfect accessibility that still buries its CTA, lies about where a link goes, or hands off to a broken booking calendar has not passed Step 6 — it has passed everything Step 6 was built to catch except the one thing that determines whether the project succeeds.

---

## Report format

```
ID: CV1 / CV2 / CV3 / CV4 / CV5 / CV6 / CV7
Severity: Blocker
Location: [page + section]
Finding: [what was approved vs. what shipped]
Root cause: [where in Step 5 the drift happened, if known]
Fix: [specific correction — reference back to the approved Step 2.3 / 4.3 document, do not improvise a new structure here]
```

---

## Why this exists as a separate file and not folded into 6.2

The AI self-audit in `6.2-audit-prompt.md` runs on pasted HTML in a fresh chat, with no access to the approved conversion architecture or layout preview documents — it can catch AI-slop patterns from the HTML alone, but it cannot know what section order or CTA placement was actually approved for this specific page. This check requires the Step 2.3 and Step 4.3 documents open side by side with the live page. Run it as its own pass, with those documents in hand — not as a mental note while reading through `6.2`'s output.
