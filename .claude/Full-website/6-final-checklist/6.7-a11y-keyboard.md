# Step 6.7 — Auditoria de Acessibilidade por Teclado

Rode esta auditoria manualmente após o audit de IA (6.2). Todos os itens são Blockers — nenhum vai para produção sem passar.

WCAG 2.1 AA cobertos: 2.1.1 (Keyboard), 2.1.2 (No Keyboard Trap), 2.4.1 (Bypass Blocks), 2.4.3 (Focus Order), 2.4.7 (Focus Visible), 4.1.2 (Name/Role/Value), 1.3.1 (Info and Relationships).

---

## Como testar

Desconecte o mouse. Use apenas teclado:
- **Tab**: avança para o próximo elemento interativo
- **Shift+Tab**: volta
- **Enter / Space**: ativa botões e links
- **Esc**: fecha modais e dropdowns
- **Setas**: navega dentro de componentes (menus, tabs, selects)

Faça o percurso completo de cada página. Não pare no primeiro problema.

## A1 — Skip link (Bypass Blocks)

**WCAG 2.4.1 — Críério A**

O que é: um link invisível no topo do HTML que permite ao usuário de teclado pular direto para o conteúdo principal, ignorando o header e a navegação.

**Teste:**
1. Abra a página
2. Pressione Tab uma vez — o primeiro foco visível deve ser um link "Pular para o conteúdo" ou similar
3. Pressione Enter — o foco deve saltar para o `<main>` ou para a âncora `#conteudo`

**Fix se falhar:**
```html
<!-- Primeiro elemento no <body> -->
<a href="#main-content" class="skip-link">Pular para o conteúdo</a>

<!-- No CSS (aparece ao receber foco) -->
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  background: #000;
  color: #fff;
  padding: 8px 16px;
  z-index: 9999;
  text-decoration: none;
}
.skip-link:focus {
  top: 0;
}

<!-- Na tag <main> -->
<main id="main-content" tabindex="-1">
```

**Grep para verificar existência:**
```bash
grep -n "skip-link\|skip-to\|pular para\|Pular para" *.html **/*.html
```

## A2 — Tab order lógico (Focus Order)

**WCAG 2.4.3 — Críério A**

O que é: a ordem de foco deve seguir a ordem visual da página. Elementos fora de ordem criam confusão de navegação.

**Teste:**
1. Comece no topo da página
2. Pressione Tab repetidamente, observando onde o foco vai
3. A sequência deve ser: skip link → navegação → hero → seções em ordem → footer

**Flags:**
- Foco salta para o rodapé antes de percorrer o conteúdo
- Foco fica preso em um elemento sem saída
- Modais abertas não capturam o foco (o foco continua atrás delas)
- Após fechar um modal, o foco não volta para o elemento que o abriu

**Fix:**
- Nunca use `tabindex` acima de 0 — reordena o foco de forma imprevisível
- Use `tabindex="0"` para tornar elementos não-nativos focáveis
- Para modais: ao abrir, mover o foco para o primeiro elemento interativo dentro da modal; ao fechar, devolver o foco ao botão que a abriu

**Grep para tabindex problemático:**
```bash
grep -rn "tabindex=\"[1-9]" *.html **/*.html
```

## A3 — Sem keyboard traps (No Keyboard Trap)

**WCAG 2.1.2 — Críério A**

O que é: o usuário nunca deve ficar preso em um componente sem conseguir sair com Tab ou Esc.

**Teste:**
1. Entre em todos os componentes que recebem foco: formulários, dropdowns, modais, sliders, embeds
2. Em cada um: tente sair com Tab (próximo), Shift+Tab (anterior), e Esc (fechar)
3. Nenhum deve travar

**Flags:**
- Modal que não fecha com Esc
- Dropdown de navegação que captura o foco indefinidamente
- iFrame ou widget de terceiro que absorve todos os Tabs
- Formulário com campo que não permite Tab para o próximo campo

**Fix para modais:**
```js
// Armadilha de foco intencional dentro da modal (correto)
modal.addEventListener('keydown', (e) => {
  if (e.key === 'Escape') closeModal();
  if (e.key === 'Tab') {
    // manter o foco dentro da modal
    const focusable = modal.querySelectorAll('button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])');
    const first = focusable[0];
    const last = focusable[focusable.length - 1];
    if (e.shiftKey && document.activeElement === first) {
      e.preventDefault(); last.focus();
    } else if (!e.shiftKey && document.activeElement === last) {
      e.preventDefault(); first.focus();
    }
  }
});
```

## A4 — ARIA labels em botões de ícone (Name/Role/Value)

**WCAG 4.1.2 — Críério A**

O que é: botões que contêm apenas um ícone (SVG, imagem, FontAwesome) precisam de nome acessível — sem ele, um leitor de tela anuncia apenas "botão" sem contexto.

**Teste:**
1. Identifique todos os botões e links que contêm apenas ícone sem texto visível
2. Inspecione cada um: deve ter `aria-label` ou `aria-labelledby` ou `<title>` dentro do SVG

**Flags:**
```html
<!-- Errado: nenhum nome acessível -->
<button><svg>...</svg></button>

<!-- Correto -->
<button aria-label="Fechar menu"><svg aria-hidden="true">...</svg></button>
<button><svg><title>Fechar menu</title>...</svg></button>
```

**Grep para botões e links sem texto nem aria-label:**
```bash
grep -n "button>\|<button " *.html **/*.html | grep -v "aria-label\|aria-labelledby"
```

Inspecionar manualmente os resultados — nem todo hit é um problema (botões com texto visível são OK).

## A5 — Labels associados a campos de formulário (Info and Relationships)

**WCAG 1.3.1 — Críério A**

O que é: todo campo de input, select, e textarea precisa de um `<label>` com `for` apontando para o `id` do campo, ou de `aria-label`/`aria-labelledby` equivalente.

**Teste:**
1. Navegue até cada formulário com Tab
2. Ao focar em cada campo, o nome do campo deve ser anunciado (leitor de tela) ou visível
3. Placeholder não substitui label — desaparece ao digitar e não é anunciado consistentemente

**Flags:**
```html
<!-- Errado: placeholder como único label -->
<input type="email" placeholder="Seu e-mail">

<!-- Correto: label explícito -->
<label for="email">E-mail</label>
<input type="email" id="email" placeholder="nome@empresa.com">

<!-- Alternativa: aria-label se label visual não for desejado -->
<input type="email" aria-label="E-mail" placeholder="nome@empresa.com">
```

**Grep para inputs sem label:**
```bash
grep -n "<input\|<select\|<textarea" assets/css/style.css *.html **/*.html
```

Verificar manualmente se cada um tem `id` correspondente a algum `<label for="">`.

## Checklist de conclusão

Todos os itens abaixo devem ser marcados antes de continuar:

- [ ] **A1**: Skip link presente e funcional — pressionar Tab na primeira vez mostra o link, Enter pula para `<main>`
- [ ] **A2**: Tab order lógico em todas as páginas — foco percorre a página na ordem visual sem saltos estranhos
- [ ] **A2**: Modais capturam foco ao abrir e devolvem ao elemento de origem ao fechar
- [ ] **A3**: Nenhum componente prende o foco — todos os modais/dropdowns fecham com Esc
- [ ] **A4**: Todos os botões de ícone têm `aria-label` ou `<title>` no SVG
- [ ] **A5**: Todos os campos de formulário têm `<label for>` ou `aria-label` — nenhum usa apenas placeholder
- [ ] Navegar o site inteiro usando apenas Tab + Enter sem mouse — conseguiu completar o percurso principal?
