# Component / Page Request: Claude Code Template

Use this template for every component or page request within a dev session. Fill in all fields. Do not skip sections. Missing context produces generic output.

One component or one page per request. If you need multiple things, send multiple requests in sequence.

## Tips

**Be specific about copy.** Paste the exact text. Claude Code will use whatever you give it verbatim. If you write "see the approved doc", it will hallucinate copy.

**Name the DS pattern.** Claude Code has the DS in context but naming the exact pattern removes ambiguity. "Feature card component" is unambiguous. "A card for the features" is not.

**One image per line.** If you list five images in a block of text, Claude Code will miss some. One line per image, with all four details on the same line.

**Describe mobile explicitly.** The DS defines breakpoints but not every layout decision at every breakpoint. If the mobile layout is not obvious from the desktop layout, describe it. A two-column feature section could stack left-on-top or right-on-top. Specify which.

## CRITICAL AI GUARDRAILS (HALLUCINATION-PROOFING)

To prevent AI hallucinations and generic "slop" output, you must strictly follow these rules:
1. **Strict Path Correctness**: Do not link to `.html` pages. All internal navigation links must target clean URLs (e.g. `/contato/`, `/sobre/`).
2. **Absolute Asset Paths**: Ensure all styles (`/assets/css/style.css`), scripts (`/assets/js/main.js`), and images (`/assets/img/...`) use absolute, root-relative paths. Never use relative paths like `../assets/`.
3. **Verbatim Content**: Copy and code generation must use the exact pasted copy. Do not change punctuation, add words, or paraphrase.
4. **Zero inline styles**: Never generate `style=""` attributes. If a one-off style is needed, create a utility class in `assets/css/style.css` and apply that class. No exceptions.
5. **Vendor prefixes**: Write prefixed version before unprefixed for every property that requires it:
   - `-webkit-backdrop-filter` before `backdrop-filter`
   - `-webkit-mask-image` before `mask-image`
   - `-webkit-appearance` before `appearance`
   - `min-height: -webkit-fill-available; min-height: 100vh;` (not just `100vh`)
6. **Semantic HTML + ARIA**: Use semantic elements (`<main>`, `<nav>`, `<header>`, `<footer>`, `<section>`, `<article>`, `<button>`) instead of generic `<div>` where a semantic element exists. Every icon-only `<button>` or `<a>` must have `aria-label`. Every `<input>` must have an associated `<label for>`. Every dropdown/accordion trigger must have `aria-expanded`. Every inline error message must have an `id` and the field must reference it via `aria-describedby`.
7. **Disabled states**: Never use `opacity` alone to signal disabled. Use the `disabled` attribute and add a tooltip or visible label explaining how to enable the action.
8. **No positive tabindex**: Never write `tabindex` with a value above 0.
9. **Locked section order**: Never reorder, merge, or drop a section from the approved conversion architecture / layout preview to make a page "flow better." If a section's fold requirement can't be met with the approved copy as given, stop and flag it back to the user instead of cutting copy, shrinking type below the DS scale, or moving the CTA to a different section.
10. **No CDN-based CSS/JS compilers in production**: Never add `<script src="https://cdn.tailwindcss.com">` or any equivalent client-side compiler. It is render-blocking and explicitly against the tool's own production guidance. Compile to a static stylesheet instead.
11. **Link text must match its real destination**: Never point a link's `href` at a different destination than what its visible text promises (e.g. text says "access the contact page," href goes to an external scheduling tool). If the promised destination doesn't exist, build it — do not silently substitute another link.
12. **Hero default state must still pass the CTA fold check**: If the hero has JS-driven variants (carousel, segment picker), verify what renders by default on load, not just what exists somewhere in the markup. The default variant is what visitors see — it must show the approved CTA, not hide it behind a different action.
13. **Conversion events must reflect actual completion, not a click, and every tag must follow the canonical naming spec**: For CTAs that hand off to an external tool (Calendly, Typeform, WhatsApp), never fire a `generate_lead` (or equivalent conversion) event on the outbound link's `onclick` alone. Wire the real completion signal (e.g. Calendly's `calendly.event_scheduled` postMessage event via inline embed, or a thank-you redirect page) and fire the conversion event there. A click-only event can be kept for funnel analysis but must be named `cta_click`, never `generate_lead`. Use `5.1-session-start-prompt.md`'s Tagging Spec (rule 24) for the exact event names, `event_category`, and `event_label` convention — do not invent your own naming per component. Every CTA-type element must carry a tag; none ships untagged.
14. **Mobile-first CSS, always**: Write the mobile layout as the base rule with no media query, then add tablet/desktop via `min-width` queries. Never write desktop as the base with `max-width` overrides for mobile — verify the built component actually looks right at 375px in its base state before checking desktop, not after.
15. **Consent on every data-collecting form**: Any form field capturing name, email, or phone needs a consent statement linking to a real, built privacy policy page. Never link a form's consent text to a page that doesn't exist yet.
16. **No AI Copy Patterns**:
   - Do not use em dashes (`—`) in copy.
   - Do not use emojis in written body text or headlines.
   - Do not use SaaS clichés ("Not only X, but Y", "Transform your business").
