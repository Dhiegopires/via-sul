# Session Start Prompt: Claude Code Opener

This prompt re-establishes context at the start of a Claude Code session. Fill in the placeholder fields and paste this as the first prompt in a new session.

---

```markdown
You are building/adjusting the front-end for [CLIENT NAME / PROJECT NAME].

### Design System Context
The design system guidelines live in `DESIGN_SYSTEM.md` in the project root. You must read and follow its tokens, typography, grid system, color variables, and constraints for all styling decisions.
- Do not use hardcoded hex values (always use the CSS variables defined in `DESIGN_SYSTEM.md`).
- Do not exceed the maximum border-radius defined in the design system.
- Spacing should follow the spacing tokens (do not use arbitrary pixel values).

### Conversion Architecture Context — non-negotiable
The approved conversion architecture and layout preview (`conversion-architecture-[cliente].md` and `layout-preview-[cliente].md`) define the exact section order, CTA placement, and CTA repetition cadence for every page. This is not a style reference — it is a locked build contract, same tier as `DESIGN_SYSTEM.md`.
- Build sections in the exact order approved in the layout preview. Do not reorder, merge, split, or drop a section because a different order "reads better" — that decision was already made and signed off before this session started.
- The primary CTA (or the approved lower-commitment secondary CTA, if the page's awareness level requires agitation first) must render inside the first viewport at 375px mobile width first and foremost, and at a common desktop width as well, exactly as verified in the layout preview's CTA fold check. Mobile is the primary check, not the secondary one — build it mobile-first (see CSS Rule 9) so this is true by default rather than something retrofitted in afterward. If implementing the approved copy pushes it below the fold, stop and flag it — do not silently shrink type past the DS scale or cut copy to force it to fit.
- Do not introduce a new CTA, link, or button anywhere in the mandatory section sequence that was not accounted for in the approved attention ratio. Every additional competing action is a conversion cost that was not signed off.

### Current Task
In this session, we are focusing on:
**[Specify the page or component here, e.g. Rebuilding the Hero section of the Homepage]**

### Development Rules

1. **Clean URLs**: Every page must live in its own folder as `index.html` (e.g., `/contact/index.html` matches `domain.com/contact/`).
2. **Root-relative Paths**: Always reference shared assets (CSS, JS, images) using root-relative paths (e.g. `/assets/css/style.css` or `/assets/js/main.js`). Never use relative paths (`../../assets/`).
3. **No Frameworks/Build Steps**: Use raw HTML5, Vanilla CSS, and modern ES6 JavaScript.
   - **Never load a CSS/JS compiler from a CDN in production** — most commonly `<script src="https://cdn.tailwindcss.com">`. This runs the entire utility-class JIT compiler in the visitor's browser on every page load: it is render-blocking (no `defer`/`async` makes it worse), and Tailwind's own documentation states it should not be used in production. If Tailwind is wanted, compile it to a static `.css` file at build time — never ship the CDN script tag. This is a Blocker, not a style preference: it directly delays the render of the CTA the whole page exists to show.
   - Never load two competing styling systems on the same page (e.g. a compiled `style.css` *and* a CDN utility-class compiler). Pick one.
4. **Performance & Lazy Loading**:
   - Use WebP images with explicit width/height declared.
   - Hero images must NOT have `loading="lazy"`. All other images below the fold must.

### CSS Rules — non-negotiable

5. **Zero inline styles**: Never use `style=""` attributes. All styling goes in `assets/css/style.css` as reusable classes. Not one exception.
6. **CSS variables only**: Never hardcode hex values, px sizes, or font names in CSS. Always use the CSS variables defined in `DESIGN_SYSTEM.md` (e.g., `var(--color-accent)`, `var(--spacing-md)`).
7. **Vendor prefixes**: For any property that requires a webkit prefix, write the prefixed version FIRST, then the standard version. Required pairs:
   - `-webkit-backdrop-filter` before `backdrop-filter`
   - `-webkit-mask-image` before `mask-image`
   - `-webkit-appearance` before `appearance`
   - `-webkit-font-smoothing` + `-moz-osx-font-smoothing` before any font smoothing
   - For `min-height: 100vh` add fallback: `min-height: -webkit-fill-available; min-height: 100vh;`
8. **Reusable classes**: Never write a CSS rule that targets only one element by ID for styling. Use semantic class names. IDs are for JS hooks and anchor targets only.
8b. **If Tailwind is compiled to a static file (not the CDN JIT compiler banned in rule 3), rebuild it every time a class not already used elsewhere in the project appears in markup.** A precompiled `tailwind.css` only contains the utility classes the build actually scanned and found in existing files — a brand-new class (e.g. the first `h-14` or `divide-x` ever used in the project) silently produces **no CSS rule and no error**. The element just renders wrong (an avatar meant to be a circle stretches into whatever aspect ratio the source image happens to have) with nothing in the console to point at the cause. Confirmed production example: a homepage team-photo section used `w-14 h-14 rounded-full object-cover` — none of those classes had appeared anywhere else in the project, the build was never rerun, and all three photos rendered as ovals instead of circles until `npm run build:css` ran and a real screenshot confirmed the fix. Rule of thumb: after any batch of edits that introduces new utility classes, rebuild before considering the task done, and verify with an actual screenshot — the markup alone will look correct either way.
9. **Mobile-first, not desktop-first — non-negotiable.** Write the base CSS rule (no media query) for the mobile viewport. Add tablet/desktop complexity only through `min-width` media queries layered on top. Never write the desktop layout as the unqualified base rule with `max-width` queries stripping it down for mobile — that is the exact pattern that caused a production hero's CTA to require a late `!important` override just to reappear on mobile, instead of working there by default. Two reasons this is mandatory, not preference: Google's mobile-first indexing has been 100% complete since July 2024 (confirmed on Google's own Search Central blog) — the mobile render is the canonical version used for ranking every site, regardless of that site's actual desktop traffic. And mobile is just over half of global web traffic — StatCounter and DataReportal's Digital 2026 report both independently put it around 51-52% as of 2026. Build for the base case first.

### HTML/ARIA Rules — non-negotiable

10. **Document language**: The `<html>` tag must always have `lang` matching the page language. Portuguese: `<html lang="pt-BR">`. English: `<html lang="en">`. Missing `lang` is a WebAIM top-6 failure — screen readers will mispronounce every word.
11. **Semantic landmarks**: Every page must have exactly one `<main>`, one `<header>`, one `<footer>`, and `<nav>` for every navigation block. No `<div>` where a semantic element exists. Never use `<div onclick>` as a button — use `<button>`.
12. **Skip link**: First element inside `<body>` must be `<a href="#main-content" class="skip-link">Pular para o conteúdo</a>`. The `<main>` must have `id="main-content"` and `tabindex="-1"`.
13. **ARIA on icon elements**: Every `<button>` or `<a>` containing only an icon (SVG, img, or icon font) must have `aria-label="..."`. The SVG inside must have `aria-hidden="true"`.
14. **ARIA on interactive state**: Every dropdown trigger, accordion header, and collapsible element must have `aria-expanded="false"` updated to `"true"` via JS when open. Every form field with an inline error message must have `aria-describedby="[error-id]"` pointing to the visible error element.
15. **Form labels**: Every `<input>`, `<select>`, and `<textarea>` must have a `<label for="...">` that matches its `id`. Never use placeholder as the only label — placeholders vanish on input and are not consistently announced by screen readers.
16. **Disabled buttons**: Never communicate a disabled state by opacity alone — this drops contrast below 4.5:1. Use the `disabled` attribute and add a visible tooltip or inline note explaining what the user must do to enable the action.
17. **No positive tabindex**: Never write `tabindex="1"` or any positive integer. Only `tabindex="0"` or `tabindex="-1"` are acceptable.

### Code Signals — non-negotiable

18. Do not leave `href="#"` or `href=""` links.
19. Do not leave `console.log` statements in production JS.
20. Do not write explanatory comments (e.g., `<!-- Main navigation bar -->`). Only comment *why*, never *what*.
21. **Link text must match its real destination.** If a link's visible text says "access the contact page," its `href` must actually go to a contact page — never point it to the same destination as a different, already-labeled button just because the real page doesn't exist yet. A mismatched link is not a placeholder, it is a false statement the visitor discovers by clicking it, and it costs the trust the rest of the page just spent copy building. If the promised destination genuinely doesn't exist yet, say so and build it — do not quietly redirect the copy's promise elsewhere.
22. **Interactive hero elements (carousels, segment pickers, tab switchers) must not hide the approved CTA in their default, on-load state.** If the hero has JS-driven variants (e.g. a rotating card carousel that swaps which buttons are visible), check what a visitor actually sees the instant the page finishes loading — not just what exists somewhere in the static HTML. If the default variant hides the primary CTA behind a different action (like an audience picker), that default variant is the real hero as far as conversion is concerned, and it must still pass the CTA fold check from the layout preview.

### Conversion Tracking — non-negotiable

23. **A click on an outbound link is not a conversion event.** When the primary CTA hands off to an external tool the site doesn't control (Calendly, Typeform, WhatsApp, etc.), firing `gtag('event', 'generate_lead', ...)` on the button's `onclick` only proves the visitor clicked — it does not prove they booked, submitted, or completed anything. That gap makes "0 leads" impossible to diagnose: it could mean nobody clicked (a page problem) or everybody clicked and dropped off inside the external tool (not a page problem, and invisible to this event). Fix per tool:
    - **Calendly**: embed the inline widget (not a plain link) and listen for its `calendly.event_scheduled` message via `window.addEventListener('message', ...)`, firing the real conversion event only then. If an inline embed isn't possible, use Calendly's redirect/`utm` + a dedicated `/obrigado/` thank-you page the booking flow returns to, and fire the event there.
    - **Any external form tool**: use its native completion webhook or redirect-on-submit callback, not a click handler on the link that opens it.
    - Keep the click event too if useful for funnel analysis, but name it distinctly (e.g. `cta_click`) and never label it `generate_lead` — that name is reserved for confirmed completions.

24. **Tagging spec — the exact event, category, and label every element gets. Do not invent your own naming per project.**

Method: `gtag.js` loaded directly in `<head>` (per the GA4 install rule above), never Google Tag Manager unless the client already has a GTM container in use elsewhere — flag it and ask before introducing GTM into a project that doesn't have it.

**Canonical event names — this fixed list only:**

| Event name | Fires on | Never use for |
|---|---|---|
| `generate_lead` | Confirmed completion only: Calendly `calendly.event_scheduled`, a form's real submit, a thank-you page load | A click on anything. Ever. |
| `cta_click` | Every click on any CTA button/link (primary or secondary), regardless of destination | Form submissions (use `form_submit`) |
| `form_submit` | The lean fallback form (or any native form) submitting successfully | External tool hand-offs (those get `cta_click` + the tool's own completion event) |
| `phone_click` | A `tel:` link | WhatsApp links (use `whatsapp_click`) |
| `whatsapp_click` | A `wa.me` or `api.whatsapp.com` link | Phone links |

**Parameter convention — every `cta_click` and `generate_lead` call carries both:**
- `event_category`: one of `cta` (main conversion actions), `nav` (header/footer navigation-area CTAs), `form` (form-related), `content` (content engagement — e.g. a gated PDF download). Do not invent a fifth category without flagging it.
- `event_label`: short, specific, `snake_case`, unique per distinct CTA instance on the page — e.g. `hero_demo`, `nav_demo`, `final_cta_calendly`, `guide_download`. The monthly report and `6.8-conversion-signals.md` CV3 (attention ratio) depend on being able to tell which specific button on which specific page drove the click — `event_label: 'cta'` on every button defeats the entire point of tracking.

**Placement rule:** every CTA-type element defined in the approved conversion architecture (Step 2.3) and layout preview (Step 4.3) must carry a tag — primary CTA, secondary CTA, nav CTA, footer CTA, any repeated CTA per the cadence rule. No CTA ships untagged; an untagged CTA is invisible to `6.8-conversion-signals.md` CV3 and to the monthly report, which defeats the reason tagging exists at all.

**Worked example (copy this pattern, don't reinvent it per component):**
```html
<a href="https://calendly.com/cliente/demo" target="_blank" rel="noopener noreferrer"
   class="btn btn-primary"
   onclick="if(window.gtag)gtag('event','cta_click',{event_category:'cta',event_label:'hero_demo'})">
  Agendar demonstração
</a>
```
Paired with the inline Calendly embed's `calendly.event_scheduled` listener (rule 23 above) firing `gtag('event','generate_lead',{event_category:'cta',event_label:'hero_demo'})` — same `event_label`, so the monthly report can connect the click to the confirmed lead for that exact CTA instance.

### Legal & Privacy — non-negotiable

25. **Every form that collects personal data needs consent and a real privacy policy to link to.** LGPD applies to any Brazilian site collecting name, email, phone, or similar. Every form must have a consent checkbox or statement ("Ao enviar, você concorda com nossa [Política de Privacidade]") linking to a real `/politica-de-privacidade/` page — not a placeholder, not a page that 404s. If the briefing master flags the privacy policy as `[PENDING]`, stop and flag it back rather than shipping a form with a broken or missing link. If the site uses cookies/analytics beyond strictly necessary ones (GA4, Hotjar, pixel trackers all count), add a cookie consent banner before those scripts fire.

Please confirm you have read `DESIGN_SYSTEM.md` and the approved conversion architecture / layout preview for this page, and are ready to start building the specified page/component in the locked section order.
```
