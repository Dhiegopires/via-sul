# Monthly Analysis: AI Execution Prompt
> Build tier only. Run this before generating the report and before filling the A/B test brief.
> Paste this into a new Claude chat with the month's raw data. Do not run in the same chat as the report.
> See the "Reading GA4 + Hotjar together" table in `README.md` for the diagnosis logic this analysis draws on.

You are analyzing one month of performance data for a VellumWire client website. Your job is to identify what the data actually shows — not what we hoped it would show — and produce one specific, actionable recommendation for the next month.

You are not writing marketing copy. You are reading data and making a call.

## WHAT TO PRODUCE

### Part 1: What the data shows

Read the data above. For each of the following, state what is actually happening — not what we want to be happening.

**Traffic:**
Is traffic growing, flat, or declining? Is the source mix healthy for this site's goal (e.g. organic for a local service business, or direct for a referral-driven business)? Is any source spiking or dropping in a way that needs attention?

**Conversion:**
What is the sessions-to-conversion rate? Is it moving in the right direction? If conversion dropped while traffic held, where is the drop-off likely occurring? If both dropped, is it a traffic quality issue or a page issue?

**Hotjar (if data provided):**
What does scroll depth tell us about whether visitors are reaching the CTA? What does the CTA click rate tell us about whether the CTA is working once they see it? Do these two numbers together point to a page structure problem or a copy problem?

**Anomalies:**
Is there anything in the data that doesn't fit the expected pattern? Flag it. Don't explain it away.

### Part 2: Diagnosis

State the most important thing the data is telling us about this site's performance right now. One sentence. Not a list.

### Part 3: Recommendation for next month

State the one action that is most likely to move the primary conversion metric based on what the data shows.

Format it as:
- **Action:** [What to do — specific]
- **Why:** [What data point drives this recommendation — cite the number]
- **Expected outcome:** [What metric should move if this works, and in which direction]
- **How to measure it:** [GA4 event or Hotjar metric to watch]

Do not recommend more than one action. If multiple things need attention, rank them and recommend the highest-priority one. The others can wait.

### Part 4: A/B test hypothesis (if applicable)

If the recommendation involves a change to copy or design that can be tested without a full redesign, generate an A/B test hypothesis.

Format:
- **Element to test:** [specific element on a specific page]
- **Control:** [current state]
- **Variant:** [proposed change]
- **Hypothesis:** If we [change X], then [metric Y] will [direction] because [reason from data].
- **Primary metric:** [the one number that decides the winner]

If the recommendation is not testable (e.g. it's a traffic-source issue or a technical fix), write "Not applicable — this recommendation does not require an A/B test."

---

## RULES FOR THIS PROMPT

- Every claim must reference a specific number from the data.
- Do not say "it seems like" or "it may be that." State what the data shows, then state what you conclude from it.
- If the data is insufficient to make a confident diagnosis, say so explicitly and state what data would be needed.
- Do not recommend actions that are outside the Build tier scope (full redesigns, new features, new identity work).
- Do not pad the recommendation with a list of ideas. One action. The right one.
